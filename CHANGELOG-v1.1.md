## 这次更新了什么？

**拍照就能结账。** 不用再在商品列表里翻找——对准商品拍一张照，系统自动识别是哪件商品，直接加入购物车。摊主出摊效率直接拉满。后端从 tract 换成 ONNX Runtime，性能翻倍，还支持 GPU / NPU 加速。默认模型切换到 FP16 量化版（17 MB），体积减半、质量几乎无损。

**更聪明的商品管理。** 每件商品可以上传多张不同角度的照片，系统会用 AI 学习它们的样子。照片越多，识别越准。新增标签系统，博丽灵梦、红色、东方 Project 等多维度快速筛选。

**摊位平板体验全面重做。** 闲置吸引屏、顾客引导、加购动画、售罄置底、低库存预警、结算确认、双收款码、支付倒计时优化——为真实漫展场景打磨的每一个细节。

**管理端上手变简单。** 首次启动有专门的欢迎页引导你进入后台配置；控制台有快速开始进度条、AI 拍照识别推荐卡和帮助气泡；全局商品库新增"识别图"列（🔴/🟡/🟢 三档），哪些商品还没准备好一眼可见。AI 视觉面板从"诊断式状态"改成"行动式引导"——告诉你"下一步该做什么"而不是"当前是什么状态"。

**界面全面翻新。** 颜色、字号、圆角、阴影全部统一到一套设计系统，暗色模式和亮色模式都更舒服。购物车按钮变大了（终于不用指甲尖去点了），库存显示变紧凑了（不再占半个屏幕）。

**数据库底层重铸。** SQLite 启用 WAL + busy_timeout + foreign_keys，连接池 PRAGMA 正确下发，补上关键索引，修复了一批并发竞争导致的库存/订单数据错误。多人同时下单不再丢库存。

**更安全更稳定。** 修复了多个安全漏洞和稳定性问题，订单状态流转有了严格校验，取消/退款链路不再吞错，路径遍历和越权漏洞全部堵上。

## 一、AI 图像嵌入与智能搜索（Vision）

本版本引入以**图像嵌入（Image Embedding）**为核心的视觉识别系统。整体逻辑分为三层：数据层（商品多图管理）→ 模型层（AI 嵌入引擎）→ 应用层（图搜图交互）。

### 1.1 商品多图管理

每个商品现在可以关联多张图片，用于 AI 识别训练。

- 编辑商品弹窗新增**「AI 视觉图库」Tab**，支持预览、添加、删除商品的视觉图片
- 一个商品支持多张不同角度/场景的图片，每张独立生成 embedding 向量
- 旧版单张主图自动同步，无需手动迁移
- 图片新增/更新后自动触发增量索引重建，无需手动操作
- **识别图可视化状态**：全局商品库列表新增"识别图"列，每个商品显示 🔴 未上传 / 🟡 1~2 张 / 🟢 3 张及以上三档状态，让"哪些商品还没准备好"一眼可见
- **批量筛选**：列表筛选栏新增"只看缺识别图的商品" checkbox，配合新增的`?filter=need_vision_images` 路由参数，可从 AI 视觉面板一键跳转并自动筛选
- **识别图快捷入口**：操作列新增"识别图"按钮，点击直接打开编辑弹窗的识别图 Tab，无需再经过"编辑→切 Tab"两步
- **未激活模型时主动警告**：打开识别图 Tab 时若 AI 模型未激活，顶部显示橙色警告"现在上传的图片会暂存但无法生成识别向量"，避免用户白费功夫上传一堆图后才发现问题
- 上传指引改为具体数字"每个商品上传 **1~3 张** 不同角度的照片"，替代原来模糊的"越多越好"

### 1.2 AI 模型配置与索引

管理员可在控制台页面管理 AI 模型和嵌入索引。

- 新增**「AI 视觉识别」管理面板**（控制台页面内）
- 支持查看运行时状态：当前模型、索引数量、最后构建时间、是否就绪
- 支持**安装/激活/删除**模型，安装过程有进度条反馈
- 支持手动触发**增量构建**或**全量重建**索引
- 默认模型从 MobileNetV3-Small 更换为 **MobileCLIP-S0**，检索质量大幅提升
- 新增 DINOv2-Small 作为可选高级模型（桌面端推荐）
- **行动导向的状态引导**：面板不再只显示诊断式状态（"未就绪"/"索引为空"），而是根据当前状况告诉用户下一步该做什么——"第 1 步：从下方可用模型中选择一个下载"、"第 2 步：点击激活按钮"、"索引为空：请给商品上传识别图，然后点击增量构建索引"等
- **去上传识别图跳转**：索引为空时 nextAction 提示内嵌跳转按钮，一键跳到全局商品库并自动筛选出缺识别图的商品
- **构建按钮 tooltip**：悬停"增量构建"/"全量重建"时显示两者区别说明，非技术用户不再盲选
- **错误态带重试**：模型列表加载失败时显示"加载失败，请刷新"+重新加载按钮，不再卡在"加载中..."
- **HelpBubble 帮助气泡**：面板标题旁新增 `?` 按钮，点击弹出该面板的 7 条使用提示（完整流程、模型选择、激活语义、构建区别、上传建议、设备选择、排障思路）
- **模型列表信息密度优化**：卡片右下角新增文件大小 chip（主色高亮），描述文案重写以体现场景定位（⭐ 默认推荐 / ⭐ 高精度推荐 / 语义理解 / 参考精度）

### 1.3 图搜图（以图搜商品）

基于嵌入向量的实时图像搜索，部署在顾客点单页面。

- 顾客页面新增**「拍照识别」模式**，可与传统商品列表模式一键切换
- **摄像头取景模式**：打开前置/后置摄像头，取景框引导对准商品，按快门自动拍照搜索
- **文件上传模式**：拖拽或点击上传图片，也可作为 fallback
- 搜索返回 Top-K 匹配结果，显示商品名称、缩略图、匹配度百分比
- 同一商品多张图片取最高匹配分，避免重复结果
- 支持**置信度判断**：匹配度过低或前两名差距过小时，标记为「置信度较低」
- 点击搜索结果自动加入购物车

## 四、Vision 推理引擎迁移

- **推理后端从 tract 切换到 ONNX Runtime**：tract 对 FP16/INT8 量化模型不支持，且缺少 CPU SIMD 优化。ONNX Runtime 是行业标准推理引擎，原生支持所有精度和算子，CPU 推理更快
- **运行时动态加载**：`onnxruntime.dll` (14MB) 随安装包分发，运行时动态加载，用户无需额外安装
- **推理性能日志**：每次推理输出预处理/推理/总耗时及平均值，方便调优
- **新增 FP16 量化模型**：
  - ConvNeXt V2 Pico FP16（17 MB）：默认推荐，纯 CNN，体积最小、速度最快
  - DINOv2-Small FP16（43 MB）：高精度推荐，自监督 ViT，对细节和角度最敏感
  - 采用 `keep_io_types=True` 的 FP16 转换策略——权重是 FP16、IO 仍是 FP32，Rust 侧推理代码零改动
  - 在 8 张合成图 + 30 张真实商品图上验证：R@1 vs FP32 = 1.0，cos_min = 0.9999+，体积减半几乎无损
- **默认模型切换**：`active_model_id` 从 `convnextv2_pico_fp32` 改为 `convnextv2_pico_fp16`，新用户首次下载只需 17 MB
- **MobileCLIP-S0 明确保留为 FP32**：实测其 FP16/INT8 变体会产生错误 embedding（cos vs FP32 仅 0.22），已在 manifest 和模型描述中明确标记"DO NOT quantize"
- **模型 registry 扩展**：`ModelManifest` 新增 `size_mb: Option<f64>` 字段，API 响应透传，前端用带主色高亮的 chip 展示（替代原本意义不大的 `model_version` chip）
- **模型面板信息密度优化**：可用模型卡片新增文件大小标识、描述文案重写以体现场景定位（⭐ 默认推荐 / ⭐ 高精度推荐 / 语义理解 / 参考精度）
- **转换脚本 + 文档化**：新增 `scripts/convert_to_fp16.py`（FP32→FP16 批量转换 + 数值校验），以及 `src-tauri/docs/model-fp16-conversion.md`（版本约束、环境规格、opset 策略、落地流程）

## 五、安全与稳定性

### 安全修复
- 修复静态文件服务的路径遍历漏洞，非法路径请求返回 403
- 修复订单取消的越权问题，摊主无法再操作其他展会的订单
- 登录密码输入框改为密码模式，支持点击显示/隐藏，防止旁人窥屏
- 登录调试日志不再打印密码明文
- 每台设备启动时自动生成独立 JWT 密钥

### 稳定性修复
- 后端所有 HTTP handler 消除 `.unwrap()` panic 风险，数据库异常时返回错误信息而非崩溃
- 删除展会改为事务操作，级联删除失败时自动回滚，不再留下不一致数据
- 订单状态更新增加白名单验证，防止无效状态写入
- 登出 Cookie 修复：HTTP 环境下退出登录不再失效
- AI 模型加载和推理不再阻塞 HTTP 服务，多人同时扫码时页面正常响应
- 多人同时触发 AI 搜索时，模型只加载一次，不再重复加载浪费内存

### 数据库底层重构
- **连接池 PRAGMA 修复**：从 `execute(&pool)` 改为 `SqliteConnectOptions`，确保 WAL/busy_timeout/foreign_keys 对每条连接生效而非仅 1 条
- **启用 WAL 模式**：读写并发不再互相阻塞
- **启用 busy_timeout=5s**：遇锁等待而非立即报错，解决点单间歇性失败
- **启用 foreign_keys=ON**：级联删除约束实际生效
- **连接池从 10 缩减为 5**：SQLite 写串行，多连接反而增加锁竞争
- **补充缺失索引**：`order_items(order_id)` 和 `orders(event_id, status)`，减少查询扫描和锁持有时间

### SQL 事务与并发安全
- **修复库存扣减并发丢失**：`SET current_stock = ?` 改为原子操作 `SET current_stock = current_stock - ?`，防止两个顾客同时下单导致库存只减一次
- **修复库存更新并发覆盖**：管理员调整库存改为原子 SQL `current_stock = current_stock + (new - initial_stock)`，不再覆盖顾客订单的扣减
- **修复 `last_insert_rowid()` 连接池不安全**：INSERT 改用 `RETURNING id`，避免从不同连接拿到错误 ID
- **修复订单创建后多余查询**：INSERT 改用 `RETURNING *`，事务内直接拿到完整订单数据
- **修复退库存错误被静默吞掉**：所有 `let _ = sqlx::query(...)` 改为错误处理和返回
- **修复已取消订单可改回**：新增状态流转校验，禁止从 cancelled 改为 pending/completed
- **修复 DELETE 错误被吞掉**：所有写操作统一加错误处理

### .boothpack 导入导出修复
- **修复导入卡死**：将阻塞性文件解压（`std::fs`）移到 `tokio::spawn_blocking`，不再阻塞 HTTP 服务
- **修复事务过长**：事务只包裹 DB 操作，文件 I/O 移到事务外
- **修复导入成功提示字段名**：前端 `count` → `products_count` / `images_count`
- **新增导入进度条**：上传/解压/写入三阶段可视化反馈

### 统计页面修复
- **修复订单数膨胀**：`COUNT(o.id)` → `COUNT(DISTINCT o.id)`，多商品订单不再被重复计数
- **修复图表跳过无销售时段**：填充首单到末单之间所有时间桶，无销售时段显示为 0

- **修复 Y 轴刻度不整**：改为 nice ticks 算法，自动取 ¥0/¥20/¥40 等整数刻度
